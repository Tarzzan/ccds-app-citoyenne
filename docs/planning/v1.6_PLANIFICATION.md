# Planification v1.6 — CCDS Citoyen

**Date :** 04 Mars 2026
**Auteur :** Manus AI

## 1. Contexte et Analyse des Lacunes

L'analyse post-v1.5 a mis en évidence que le projet a atteint une maturité fonctionnelle significative. Les lacunes restantes ne sont plus des fonctionnalités de base, mais des améliorations stratégiques axées sur l'engagement communautaire, l'intelligence des données et l'ouverture de la plateforme.

Les axes principaux identifiés sont :

1.  **Engagement Communautaire :** Le système actuel est transactionnel (signalement → résolution). Il manque des outils pour créer une véritable communauté d'utilisateurs actifs.
2.  **Intelligence des Données (Data Intelligence) :** L'application collecte une grande quantité de données géolocalisées et catégorisées, mais l'exploitation reste basique (statistiques descriptives).
3.  **Interopérabilité et Ouverture :** L'application fonctionne en silo. Pour maximiser son impact, elle doit pouvoir s'intégrer avec d'autres systèmes (API publique, webhooks).
4.  **Tests et Qualité :** Il n'existe aucun test End-to-End (E2E), ce qui rend les régressions visuelles et fonctionnelles possibles à chaque nouvelle version.

## 2. Tickets Priorisés pour la v1.6

La v1.6 se concentrera sur ces 4 axes avec 10 tickets priorisés.

### Thème 1 : Engagement Communautaire (P1 - Important)

| Ticket | Titre | Description | Valeur Ajoutée |
| :--- | :--- | :--- | :--- |
| **UX-10** | **Sondages & Consultations** | Permettre aux administrateurs de créer des sondages géolocalisés ("Êtes-vous favorable à un nouveau parc ici ?") auxquels les citoyens peuvent répondre. | Impliquer les citoyens dans les décisions locales, augmenter l'engagement. |
| **UX-11** | **Mentions `@` dans les commentaires** | Permettre de mentionner un autre utilisateur (citoyen ou agent) dans un commentaire pour le notifier. | Faciliter la communication directe et la collaboration. |
| **UX-12** | **Événements communautaires** | Permettre aux agents de créer des événements (nettoyage de quartier, réunion publique) auxquels les citoyens peuvent s'inscrire. | Transformer l'application en un outil de coordination locale. |

### Thème 2 : Intelligence des Données (P1 - Important)

| Ticket | Titre | Description | Valeur Ajoutée |
| :--- | :--- | :--- | :--- |
| **ADMIN-09** | **Dashboard Admin Temps Réel** | Rendre le tableau de bord admin dynamique via WebSockets (nouvelles inscriptions, signalements, votes en direct). | Donner aux agents une vision vivante de l'activité citoyenne. |
| **ADMIN-10** | **Analyse Prédictive** | Analyser l'historique pour prédire les futures zones à risque (ex: accumulation de dépôts sauvages à certains endroits/périodes). | Passer d'une gestion réactive à une gestion proactive des problèmes. |
| **ADMIN-11** | **Export RGPD des données citoyen** | Permettre à un citoyen de demander et recevoir une archive complète de ses données personnelles (profil, signalements, votes, etc.). | Conformité légale (RGPD) et renforcement de la confiance. |

### Thème 3 : Interopérabilité (P2 - Normal)

| Ticket | Titre | Description | Valeur Ajoutée |
| :--- | :--- | :--- | :--- |
| **API-02** | **API Publique (lecture seule)** | Exposer une API publique documentée (avec Swagger UI) pour permettre à des tiers (médias, chercheurs) d'accéder aux données anonymisées. | Transparence, réutilisation des données, innovation ouverte. |
| **API-03** | **Webhooks sortants** | Permettre aux administrateurs de configurer des webhooks pour notifier des systèmes externes lors d'événements (nouveau signalement, changement de statut). | Intégration avec des outils tiers (Slack, Zapier, systèmes de GMAO). |

### Thème 4 : Qualité et Tests (P0 - Critique)

| Ticket | Titre | Description | Valeur Ajoutée |
| :--- | :--- | :--- | :--- |
| **TEST-03** | **Tests End-to-End (E2E) avec Detox** | Mettre en place des tests E2E pour les flux critiques de l'application mobile (inscription, création de signalement, vote). | Garantir l'absence de régression sur les parcours utilisateurs clés. |
| **TECH-04** | **API Rate Limiting par IP et par utilisateur** | Affiner le rate limiting pour éviter les abus tout en étant plus flexible pour les utilisateurs légitimes. | Sécurité et disponibilité de l'API. |

## 3. Ordre de Développement Recommandé

1.  **TEST-03 + TECH-04** : Fondations de qualité et de sécurité.
2.  **ADMIN-11** : Conformité RGPD, un prérequis légal.
3.  **UX-10 + UX-11 + UX-12** : Bloc "Engagement Communautaire".
4.  **ADMIN-09 + ADMIN-10** : Bloc "Intelligence des Données".
5.  **API-02 + API-03** : Bloc "Interopérabilité" en dernier, une fois les données stabilisées.

## 4. Migrations SQL Prévisionnelles

```sql
-- Pour UX-10 : Sondages
CREATE TABLE polls (
  id INT AUTO_INCREMENT PRIMARY KEY,
  title VARCHAR(255) NOT NULL,
  description TEXT,
  created_by INT NOT NULL, -- admin_id
  location GEOMETRY,
  start_date DATETIME,
  end_date DATETIME,
  created_at DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP,
  FOREIGN KEY (created_by) REFERENCES users(id)
);

CREATE TABLE poll_options (
  id INT AUTO_INCREMENT PRIMARY KEY,
  poll_id INT NOT NULL,
  option_text VARCHAR(255) NOT NULL,
  FOREIGN KEY (poll_id) REFERENCES polls(id) ON DELETE CASCADE
);

CREATE TABLE poll_votes (
  id INT AUTO_INCREMENT PRIMARY KEY,
  poll_option_id INT NOT NULL,
  user_id INT NOT NULL,
  created_at DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP,
  UNIQUE KEY uq_poll_user (poll_option_id, user_id),
  FOREIGN KEY (poll_option_id) REFERENCES poll_options(id) ON DELETE CASCADE,
  FOREIGN KEY (user_id) REFERENCES users(id) ON DELETE CASCADE
);

-- Pour UX-12 : Événements
CREATE TABLE events (
  id INT AUTO_INCREMENT PRIMARY KEY,
  title VARCHAR(255) NOT NULL,
  description TEXT,
  location VARCHAR(255),
  event_date DATETIME NOT NULL,
  created_by INT NOT NULL, -- admin_id
  created_at DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP,
  FOREIGN KEY (created_by) REFERENCES users(id)
);

CREATE TABLE event_rsvps (
  id INT AUTO_INCREMENT PRIMARY KEY,
  event_id INT NOT NULL,
  user_id INT NOT NULL,
  status VARCHAR(20) NOT NULL, -- 'attending', 'interested'
  created_at DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP,
  UNIQUE KEY uq_event_user (event_id, user_id),
  FOREIGN KEY (event_id) REFERENCES events(id) ON DELETE CASCADE,
  FOREIGN KEY (user_id) REFERENCES users(id) ON DELETE CASCADE
);

-- Pour API-03 : Webhooks
CREATE TABLE webhooks (
  id INT AUTO_INCREMENT PRIMARY KEY,
  target_url VARCHAR(255) NOT NULL,
  event VARCHAR(100) NOT NULL, -- e.g., 'incident.created', 'incident.status.changed'
  is_active TINYINT(1) NOT NULL DEFAULT 1,
  created_at DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP
);
```
